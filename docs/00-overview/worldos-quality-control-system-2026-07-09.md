# WorldOS 质量把控体系

> [!IMPORTANT]
> 本文档用于约束“世界 / 宇宙体验”后续阶段的验收。它强调真实体验、真实浏览器、真实本地构建和真实证据，避免用旧构建、旧截图或纯脚本通过来替代质量。

## 1. 质量目标

质量目标不是“命令都通过”，而是：

**任何人在本地或局域网打开 WorldOS，都能看到可运行、可阅读、可降级、可探索的个人数字世界。**

## 2. 质量分层

| 层级 | 关注点 | 不通过表现 |
| --- | --- | --- |
| P0 可运行 | build、lint、白屏、路由、LAN 访问 | 白屏、构建失败、关键路由 500 |
| P1 可读 | 中文优先、静态降级、正文密度、移动阅读 | 只有骨架、文字遮挡、移动端不可读 |
| P2 可感知 | 首访仪式、场景人格、转场连续性 | 像普通博客、动效弱到不可感知 |
| P3 可探索 | Atlas、Timeline、Archive、Paths、Node 互通 | 页面孤岛、路径像链接清单 |
| P4 可治理 | 权限、内容事实源、RC 证据、报告策略 | 前端硬编码权限、证据不可复现 |

## 3. 必跑门禁

| 门禁 | 推荐命令 | 目的 |
| --- | --- | --- |
| 文档注册 | `npm run check:docs` | 确认关键文档被系统识别 |
| 类型 / 语法 | `npm run lint` | 捕获基础代码问题 |
| 构建 | `npm run build` | 确认 fresh build 可用 |
| 严格检查 | `npm run check:strict` | 运行项目主干质量检查 |
| 本地 RC | `npm run release:local-rc` | 生成可信本地 / LAN 验收 |
| 浏览器 QA | Playwright CLI 或项目内 browser smoke | 验证真实页面不是白屏 |

> [!NOTE]
> 当前请求阶段只补文档，不执行功能开发；具体命令在后续实现阶段按变更范围执行。

## 4. 世界体验人工验收

脚本只能证明“没有明显故障”，不能证明“像世界”。后续每个体验阶段必须进行人工验收。

| 场景 | 人工验收问题 | 失败判定 |
| --- | --- | --- |
| Home | 第一眼是否像进入世界入口？ | 仍像 Hero + 卡片列表 |
| Atlas | 是否能看出区域、地点、关系和方向？ | 像分类页或文章网格 |
| Timeline | 是否有时间流动和回看感？ | 像普通时间线列表 |
| Archive | 是否像档案馆 / 记忆库？ | 像标签归档页 |
| Paths | 是否像旅程？ | 像链接清单 |
| Node | 是否像抵达一个地点？ | 像普通文章详情 |
| Lighthouse | 是否清楚表达只读、可解释、不可越权？ | 像普通聊天入口 |

## 5. 视觉 QA 标准

每次核心体验变更至少保留以下证据：

- desktop 截图：首屏、Atlas、Timeline、Archive、Paths。
- mobile 截图：首屏、移动导航、核心 CTA、状态卡。
- reduced-motion 截图：确认不是空白，也没有遮挡。
- fixed overlay 报告：确认固定层不遮挡主内容。
- 白屏检查：确认 root、关键标题、核心导航存在。

截图报告必须明确区分：

- `desktop`
- `mobile`
- `reduced-motion`
- `local`
- `LAN`

## 6. 动效质量边界

动效不是越多越好。每个动效必须满足：

- 语义明确：表达进入、迁移、抵达、回看或沉淀。
- 可降级：尊重 `prefers-reduced-motion`。
- 可中断：不会阻塞阅读和导航。
- 可维护：通过统一注册表、场景契约或共享工具组织。
- 可测量：至少能通过截图、DOM 状态或人工清单验收。

禁止：

- 为每个页面写一次性动效。
- 用大面积模糊和深色遮罩掩盖内容不足。
- 把核心信息藏在动画完成之后。
- 让移动端必须等待长动画才能阅读。

## 7. 性能预算

| 项 | 目标 |
| --- | --- |
| 首屏 JS | 动态世界按需加载，不阻塞静态阅读 |
| 首次进入动画 | 默认不超过 2.5 秒，允许跳过 |
| 页面转场 | 常规切换控制在 300-600ms |
| reduced-motion | 只保留必要状态变化 |
| 3D / WebGL | 仅在具体场景收益明确时引入 |
| 图谱布局 | 数据量小优先 SVG / CSS，大图再考虑 WebGL 图谱 |

## 8. 权限与内容边界

质量门禁必须覆盖权限准则：

- 后端或数据契约决定权限事实。
- 前端只负责展示、隐藏和解释，不负责制造权限结论。
- 私密、半公开、公开、vault 的边界必须来自统一模型。
- AI 灯塔只能读取公开事实源，不暗示可读取私密内容。

## 9. 证据策略

需要提交的证据：

- 关键文档。
- 质量门禁摘要。
- 当前阶段必要的稳定截图。
- 手工验收结论。

不建议提交的证据：

- 每次运行生成的大量临时截图。
- 与阶段无关的旧报告。
- 机器时间戳导致持续变动的噪声文件。

## 10. 多轮确认

后续每个实现阶段结束必须至少三轮确认：

1. **工程确认**：build、lint、check、RC。
2. **浏览器确认**：desktop、mobile、reduced-motion、LAN。
3. **体验确认**：是否满足场景人格，是否仍像骨架。

只有三轮都通过，才能标记阶段完成。

> 制定日期：2026-07-08

---

## 一、执行摘要

古月浮屿是一个以"世界模型"为核心的个人数字世界系统。经过 Phase 1-12 的持续建设，已具备完整的五层数据模型（Node -> Area -> Relation -> Path -> Event）、三层权限守门（middleware + API + 前端体现）、AI 灯塔搜索问答、一键导出传承管线、261 个治理脚本构成的门禁体系。

当前世界处于"可走且值得停留"的阶段：100 个公开节点、163 条关系星线、15 条主题路径、33 个世界事件，内容平均 745 字符，`release:local-rc` 全部通过（22 HTTP + 20 browser checks）。

本规划在此基础上，定义未来 6-12 个月的设计开发方向，从终极目标到具体执行项，层层展开。

## 三、外部对标分析

WorldOS 的定位是"个人数字世界系统"，需要跨多个赛道对标学习。以下从项目、工具生态、独特定位三个层面展开。

### 3.1 同类项目对标

| 项目 | 定位 | WorldOS 借鉴 | WorldOS 差异 |
|---|---|---|---|
| **Quartz** | 静态站点生成器，双链笔记 | 双链、图谱可视化、静态导出 | Quartz 侧重笔记，WorldOS 侧重"世界模型" |
| **Astro** | 内容站点框架 | 内容集合 (Content Collections)、岛屿架构 | Astro 通用，WorldOS 领域化 |
| **Obsidian Publish** | 笔记发布平台 | 图谱、反向链接、标签体系 | Obsidian 依赖桌面客户端，WorldOS 全 Web |
| **Logseq** | 大纲式知识管理 | 块引用、每日日志 | Logseq 无渲染层，WorldOS 有完整前端 |
| **Maggie Appleton 个人站** | 数字花园典范 | 播种-培育-收获阶段标签、认知徽章 | Maggie 手工制作，WorldOS 系统化生成 |
| **Andy Matuschak 站** | 常青笔记 (Evergreen Notes) | 概念原子化、逐步显现 | 常青注重认知，WorldOS 注重世界建构 |
| **11ty (Eleventy)** | 极简静态生成 | 零配置、生成速度 | 11ty 无 React，WorldOS 需要交互 |
| **Notion Sites** | 数据库驱动站点 | 数据库视图、模板 | Notion 闭源，WorldOS 开源 + 本地 |
| **Tana** | AI 原生知识管理 | Supertag、AI 增强 | Tana 商业，WorldOS 免费本地 |
| **RoamResearch** | 双向链接鼻祖 | 反向链接算法 | Roam 弃维，WorldOS 现代技术栈 |

**核心洞察**：现有项目大多是"笔记工具"或"内容站点"，WorldOS 独有"世界模型 + 权限分层 + 生命档案 + 传承协议"的四维定位。

### 3.2 工具生态对标

| 领域 | 当前选择 | 替代方案 | 决策 |
|---|---|---|---|
| **搜索** | Fuse.js | FlexSearch、Pagefind、MiniSearch | 保留 Fuse，M14 评估 Pagefind (静态构建友好) |
| **动效** | Framer Motion + GSAP | MotionOne、Anime.js | 保留双引擎，因职责已分离 (FM 交互、GSAP 入场) |
| **内容** | 手写 TS 数据 | Contentlayer、Velite、MDX | M13 评估 Velite (类型安全的 MDX) |
| **PWA** | 手写 Service Worker | Serwist、next-pwa | M15 评估 Serwist (Next 15 原生支持) |
| **AI SDK** | 自研 provider 抽象 | Vercel AI SDK、LangChain.js | M16 评估 Vercel AI SDK (与 Next 生态无缝) |
| **可观测** | 自研门禁 + 日志 | Sentry、PostHog | M15 集成 Sentry (错误监控) |
| **部署** | 未部署 | Vercel、Netlify、Cloudflare Pages | M15 选 Vercel 或 Cloudflare Pages |

### 3.3 WorldOS 独特定位

WorldOS 不追求成为"另一个 Quartz"，而是探索**个人数字世界**这一新品类：

1. **世界模型**：不是笔记，而是节点+关系+路径+事件构成的活体世界
2. **权限分层**：public / permission / owner-only 三层，后端控制，前端表达
3. **门禁驱动**：124 个自动化检查脚本作为世界宪法
4. **AI 边界化**：AI 是助手不是主人，可关可开可审计
5. **生命档案**：出生证明、路线、事件构成个人数字生命
6. **传承协议**：Owner-only 导出+传承机制，为长期演进准备
7. **中文优先**：默认中文，符合中文创作者认知
8. **世界互联**：Phase 12 已探索联邦协议，未来可跨世界互访

## 四、SWOT 分析

### 4.1 优势 (Strengths)
- **完整体系**：Phase 1-12 全部就绪，从数据模型到部署监控，从内容到治理
- **门禁厚重**：124 个自动化脚本，日常/边界/发布三层门禁全绿
- **权限规范**：owner/permission/public 三层，后端 gate 主导
- **AI 边界化**：`AI_ENABLED` 开关默认关闭，成本可控
- **中文原生**：文档、UI、数据全中文，无翻译负担
- **数据 schema 化**：Zod 严格校验，事件+关系+路径+节点全类型安全
- **传承协议**：出生证明+导出管线+联邦探索三管齐下
- **性能预算**：MotionConfig 全局降级，首页 SSG，构建 <30MB

### 4.2 劣势 (Weaknesses)
- **内容规模有限**：100 节点相比 Maggie Appleton (300+) 还有距离
- **技术债存量**：`_legacy/` 保留大量遗留，虽隔离但增加认知负担
- **活跃库冗余**：120 个 `src/lib/*` 待审计，可能存在功能重叠
- **无部署**：仅本地/局域网，未上线，无外部反馈闭环
- **AI 未真接**：抽象层已备好，但未接入真实 provider
- **测试深度**：Smoke test 覆盖 HTTP/browser，缺单元测试和集成测试
- **移动端体验**：响应式已做，但未做深度移动端优化
- **无可视化编辑**：所有内容需手写 TS，非技术用户门槛高

### 4.3 机会 (Opportunities)
- **AI 原生知识**：Tana、Reflect 等证明 AI + 知识管理有市场
- **数字花园复兴**：Maggie/Andy 等推动，社区对"个人世界"有认知
- **中文创作者**：中文优先的知识产品稀缺，有先发优势
- **联邦化协议**：ActivityPub 复兴，个人站点互联的时代窗口
- **本地优先浪潮**：Local-first 运动 (Ink&Switch) 与 WorldOS 理念契合
- **Next 15 稳定**：App Router 生态成熟，SSG/SSR/ISR 全就绪
- **WebGPU 图谱**：未来可用 WebGPU 渲染大型知识图谱

### 4.4 威胁 (Threats)
- **依赖漂移**：React 19 生态尚未完全稳定，可能引入兼容问题
- **过度工程**：门禁脚本 124 个，维护成本高，需警惕"仪式化"
- **技术债累积**：不做技术收束，legacy 会像雪球
- **单人维护**：无社区，若停维护则死亡，需传承协议保底
- **AI 成本失控**：未来接入真 provider 需严控成本
- **隐私合规**：若上线需考虑数据合规（备案、GDPR 等）
- **搜索引擎不友好**：SPA 特性 + 权限门槛，SEO 挑战大

## 五、终极目标

**让 WorldOS 成为可上线、可维护、可传承、可互联的个人数字世界产品。**

四个"可"字对应四个维度：

1. **可上线**：从局域网走向公网，成为一个真实可访问的个人品牌载体
2. **可维护**：技术债收束到可持续水平，单人也能长期演进
3. **可传承**：出生证明+传承协议完备，即使停维护也可移交/归档
4. **可互联**：联邦协议落地，与其他个人世界建立跨站连接

**衡量指标**：
- 内容规模：200+ 节点、300+ 关系、50+ 事件
- 上线状态：公网可访问，域名 + HTTPS + SEO 就绪
- 维护成本：日常门禁 <5min，Phase 门禁 <30min
- 用户体验：Lighthouse >= 95，移动端可用性 >= 90
- AI 体验：至少 1 个真实 provider 接入，边界完备
- 联邦互联：至少 1 个跨世界演示

## 六、里程碑规划 (M13 - M18)

六个里程碑对应六个 Phase，各自承担一个终极目标的子维度。里程碑之间可并行但存在依赖关系。

### M13：内容生态成熟 (Content Ecosystem)
- **目标**：世界内容从 100 节点扩展到 200+，形成成熟内容生态
- **交付**：200 节点、300 关系、20 路径、50 事件、100 唯一标签
- **门禁**：所有节点通过 `data:audit`；关系密度 >= 1.5；孤岛率 < 5%
- **依赖**：无（可立即启动）
- **周期估计**：4-6 周

### M14：技术债务收束 (Debt Reduction)
- **目标**：活跃库文件从 120 收束到 <80，脚本从 160 收束到 <100
- **交付**：库文件依赖图、废弃候选列表、统一工具函数库
- **门禁**：`npm run check:daily` 时长 < 3min；重复代码率 < 5%
- **依赖**：M13 内容稳定后启动，避免同时改内容和结构
- **周期估计**：3-4 周

### M15：上线就绪 (Production Ready)
- **目标**：具备公网部署能力，SEO/监控/备案就绪
- **交付**：Vercel/Cloudflare 部署、域名、HTTPS、SEO metadata、Sentry
- **门禁**：Lighthouse Performance >= 95；SEO >= 90；无控制台错误
- **依赖**：M14 技术收束（部署前减小体积）
- **周期估计**：2-3 周

### M16：AI 全光 (AI Full Beam)
- **目标**：至少一个真实 AI provider 接入并稳定运行
- **交付**：OpenAI/Anthropic/本地模型任一接入，边界完备，成本可观测
- **门禁**：AI 请求 100% 有 budget guard；错误率 < 1%；日成本可查询
- **依赖**：M15 部署完成（AI 通常需 serverless）
- **周期估计**：3-4 周

### M17：体验精修 (UX Polish)
- **目标**：达到 WCAG 2.1 AA 无障碍标准，移动端体验优化
- **交付**：无障碍审计报告、键盘导航完备、移动端专门优化
- **门禁**：axe-core 0 严重问题；触屏可用性 >= 90
- **依赖**：M13-M16 后进行整体打磨
- **周期估计**：3-4 周

### M18：世界互联 (World Federation)
- **目标**：联邦协议 (Phase 12 已探索) 落地，实现跨世界互联
- **交付**：Federation API v1、至少 1 个跨世界演示、协议开源
- **门禁**：Federation 协议通过外部验证；跨站访问延迟 < 500ms
- **依赖**：M15 上线（无公网无联邦）
- **周期估计**：4-6 周

## 八、交叉关注点 (Cross-Cutting Concerns)

跨 Phase 需要持续投入的横向议题：

### 8.1 权限体系
- 所有新功能默认走 owner-gate（若未明确说明是公开）
- 前端只做"表达"，后端做"决策"，永不硬编码权限
- 每 Phase 增补 permission audit（`scripts/permission-audit.mjs`）

### 8.2 数据完整性
- 所有节点必须通过 Zod schema
- 关系必须双向验证（源和目标都存在）
- 每次数据变更后运行 `data:audit` 全套

### 8.3 门禁演化
- 门禁是"世界宪法"，不轻易改动
- 新增门禁需通过 ADR 记录动机
- 门禁失败绝不 skip，除非 ADR 明确豁免

### 8.4 中文优先
- UI 文案默认中文，仅英文文案需要标注
- 变量/函数英文，注释和文档中文
- 日期格式统一 `YYYY-MM-DD` 或"YYYY年M月D日"

### 8.5 AI 边界化
- `AI_ENABLED` 环境变量始终存在
- AI 每次调用必有 budget guard
- AI 失败必须优雅降级到无 AI 路径

### 8.6 传承准备
- 每 Phase 更新出生证明和 CODEMAP
- 关键决策留 ADR
- 每月一次完整备份 (`backup:full`)

## 十一、执行顺序建议

考虑依赖关系和边际收益，建议执行顺序：

### 阶段 A：内容优先（M13 单干）
- **周期**：4-6 周
- **理由**：内容是世界的"实体"，先厚实实体再优化壳
- **产出**：200+ 节点世界

### 阶段 B：技术收束（M14 单干）
- **周期**：3-4 周
- **理由**：上线前必须减小体积、减少故障面
- **产出**：可维护的技术栈

### 阶段 C：上线部署（M15 单干）
- **周期**：2-3 周
- **理由**：M15 依赖 M14 完成
- **产出**：公网可访问的 WorldOS

### 阶段 D：并行三线（M16 + M17 + M18）
- **周期**：4-6 周
- **理由**：M16 AI、M17 体验、M18 联邦相互解耦，可并行
- **产出**：功能完备的成熟产品

**总时长估计**：13-19 周（约 3-5 个月）

**关键决策点 (Go / No-Go)**：
1. **M13 结束**：若节点数 < 180，延期或降低目标
2. **M14 结束**：若门禁时长 > 5min，暂停上线
3. **M15 结束**：若 Lighthouse < 80，回滚域名
4. **M16 结束**：若 AI 成本 > 预算 2x，关闭 AI 开关

# WorldOS 世界体验总控文档

> [!NOTE]
> 本文档是“静态博客 + 动态世界 / 宇宙”双形态的最高层体验总控。它不替代阶段计划，而是定义后续阶段必须共同服务的目标、边界、场景人格和验收口径。

## 1. 结论

当前项目已经具备静态内容站、动态组件、首访仪式、场景转场和本地 RC 门禁的基础能力，但还没有达到“真格世界 / 宇宙”的最终体验目标。

后续目标不能再定义为“增加动效”或“补几个组件”，而应定义为：

**让访问者在 localhost / LAN IP 中进入一个可探索、可迁移、可回看、可阅读、可沉淀的个人数字世界。**

## 2. 双形态最终目标

| 形态 | 最终目标 | 用户感知 | 成功标准 |
| --- | --- | --- | --- |
| 静态形态 | 一个长期可读、可索引、可降级、可归档的个人知识与记忆站 | 像一本组织良好的个人世界书 | 关闭 JS 或弱网下仍能阅读核心内容 |
| 动态形态 | 一个具有场景、空间、转场、关系和仪式感的个人数字世界 | 像进入 Atlas、Timeline、Archive、Paths 等不同空间 | 用户能感到进入、移动、抵达、回看和继续探索 |

静态形态负责可靠性，动态形态负责世界感。两者必须共享内容事实源，不允许形成两套内容宇宙。

## 3. 高目标定义

### 3.1 北极星

WorldOS 不是“带动效的博客”，而是一个个人数字世界操作系统：

- 内容是世界节点，不只是文章卡片。
- 区域是空间，不只是分类。
- 时间是河流，不只是列表。
- 路径是旅程，不只是链接集合。
- 档案是记忆库，不只是归档页。
- AI 是灯塔，不是太阳；只读解释、推荐和问路，不越权改写世界。

### 3.2 用户体验句子

首次访问者应能自然说出：

> 我不是打开了一个博客首页，而是进入了一个人的世界；我知道自己在哪里，可以沿路径继续探索，也能回到地图或档案馆。

## 4. 场景人格矩阵

| 场景 | 空间隐喻 | 必须具备的体验 | 禁止退化成 |
| --- | --- | --- | --- |
| Home | 世界入口 / 港口 | 首次进入仪式、当前世界状态、下一步入口 | Hero + 卡片列表 |
| Atlas | 星图 / 群岛地图 | 区域、节点、关系、距离、推荐方向 | 分类页或文章网格 |
| Timeline | 时间河 | 年月流动、事件层级、回看、继续阅读 | 时间线列表 |
| Archive | 档案馆 / 记忆库 | 分区、检索、密度、年代感、可归档 | 标签归档页 |
| Paths | 旅程系统 | 起点、进度、下一步、返回地图、完成感 | 链接清单 |
| Node | 地点 / 房间 | 节点上下文、关系、来源、后续出口 | 普通文章详情页 |
| Lighthouse | 观测站 / 只读向导 | 解释公开内容、推荐路径、说明限制 | 聊天工具页 |

## 5. 场景迁移语法

动态世界的转场不是页面淡入淡出，而是“场景迁移”。

| 迁移 | 语义 | 体验要求 |
| --- | --- | --- |
| Home -> Atlas | 从入口进入地图 | 有进入世界的方向感，展示目标空间预告 |
| Atlas -> Node | 从地图抵达地点 | 保留来源区域或关系线索 |
| Node -> Paths | 从地点进入旅程 | 明确当前节点为什么属于这条路径 |
| Paths -> Timeline | 从旅程回看时间 | 将进度转化为时间位置 |
| Timeline -> Archive | 从时间进入记忆库 | 有沉淀、归档、检索感 |
| 任意 -> Home | 返回入口 | 不像浏览器后退，而像回到世界中心 |

## 6. 文档体系

后续要真正达到目标，至少需要以下文档共同约束：

| 文档 | 作用 | 状态 |
| --- | --- | --- |
| 世界体验总控文档 | 定义最高目标、双形态、场景人格 | 已建立 |
| 质量把控体系文档 | 定义体验、工程、性能、权限、RC 门禁 | 已建立 |
| 技术栈与开源调研文档 | 定义可选工具、引入条件和否决条件 | 已建立 |
| 场景制作矩阵 | 将每个场景拆为入口、主体、转场、降级、检查项 | 待细化 |
| 转场编舞规范 | 定义首次进入、场景切换、抵达、回退、降级 | 待细化 |
| 内容生命契约 | 定义节点、区域、关系、生命周期、推荐路径 | 已有基础，需升级 |
| 本地 RC 可信化规范 | 定义 fresh build、LAN smoke、截图证据、audit 摘要 | 已有基础，需持续维护 |

## 7. 阶段规划

### M0：治理与文档锁定

目标：先统一目标和验收语言，避免继续堆“看起来完成但体验不达标”的阶段。

完成标准：

- 总控、质量、技术调研文档进入注册表。
- 后续每个阶段必须引用总控目标。
- 新功能必须说明服务哪个场景人格。

### M1：场景制作矩阵

目标：把 Home、Atlas、Timeline、Archive、Paths、Node、Lighthouse 都拆成可生产规格。

完成标准：

- 每个场景有空间隐喻、视觉骨架、动态层、内容层、降级层。
- 每个场景有“不像骨架”的人工验收标准。

### M2：首次进入与世界入口

目标：让第一次进入不再像打开网页，而像抵达世界入口。

完成标准：

- 首访仪式有入口、展开、抵达、跳过和 reduced-motion 版本。
- 首页首屏不是普通博客 Hero。

### M3：Atlas 世界地图化

目标：Atlas 成为世界主入口，而不是内容分类页。

完成标准：

- 节点空间分布、区域、关系、推荐方向可感知。
- 内容节点像地点，有进入和返回语义。

### M4：Timeline / Archive / Paths 场景人格化

目标：三类核心内容场景分别具备时间河、档案馆、旅程系统的体验。

完成标准：

- Timeline 不再是时间列表。
- Archive 不再是标签归档。
- Paths 不再是链接集合。

### M5：内容生命系统升级

目标：让场景吸收同一份内容事实源。

完成标准：

- 节点必须具备区域、关系、摘要、生命周期、路径出口。
- 无摘要、无区域、无关系的内容不能进入精选或世界地图核心层。

### M6：本地 / LAN 世界 RC

目标：保持暂不上线，只在本地与局域网形成可信验收闭环。

完成标准：

- `npm run release:local-rc` 是唯一可信入口。
- RC 包含 fresh build、LAN smoke、权限边界、截图证据、体验 QA、audit 摘要。

## 8. 不做什么

- 暂不考虑外部 Preview / Production。
- 暂不为了“宇宙感”盲目引入全站 3D。
- 暂不把权限判断硬编码到前端。
- 暂不建立第二套动态内容数据源。
- 暂不为了截图好看牺牲静态可读性与加载速度。

## 9. 通过标准

只有同时满足以下条件，才能说“真格世界”目标进入达标区：

- 首屏第一眼不像普通博客骨架。
- Atlas / Timeline / Archive / Paths 每个场景有明显人格差异。
- 路由切换能表达场景迁移，而不是普通页面进入。
- 内容节点在多个场景中被同一事实源吸收。
- reduced-motion、弱设备、静态阅读都仍然可用。
- 本地 / LAN RC 能稳定复现，并产生可信证据。

# WorldOS / 古月浮屿 全局未来规划

> 规划日期：2026-07-07
> 基线：WorldOS v1 RC8 / World Kernel 收束期
> 当前状态：`productionLive: false` / `releaseReady: false` / `cleanProductionReady: false`

---

## 一、项目定位与初衷

古月浮屿不是博客，不是作品集，不是知识库。它是一个正在收束为产品形态的**个人数字世界**：

| 方向 | 含义 |
|---|---|
| 对外 | 可进入、可探索、可停留的公开世界 |
| 对内 | 可维护、可记录、可整理的创世台 |
| 对未来 | 可导出、可回望、可传承的生命档案 |
| 对 AI | 可读、可审计、可边界化协作的世界协议 |

核心准则：降低门槛、提高体验、优先中文；后端控制权限，前端控制体现；高内聚、低耦合、模块化、页面化。

## 二、外部对比与学习

### 2.1 同类项目对比

| 项目 | Stars | 技术栈 | 定位 | 与 WorldOS 的差异 |
|---|---|---|---|---|
| Quartz | 12,696 | 自研 SSG + Obsidian | 数字花园发布器 | 专注 Obsidian 笔记发布，无世界模型、无权限分层、无 AI 边界 |
| Astro | 60,765 | 自研 SSG | 通用静态站点生成器 | 通用框架，不提供内容世界化、路径引导和节点护照 |
| Maggie Appleton 数字花园 | 350 | Next.js + MDX（已退役） | 个人数字花园 | 概念先驱，但无产品化、无门禁体系、无权限边界 |
| TuanManhCao/digital-garden | 809 | Obsidian + Quartz | 免费 Obsidian Publish 替代 | 聚焦发布，不含创世台、私密层和生命档案 |
| Logseq | 开源 | Clojure + React | 本地优先知识管理 | 本地知识图谱，不面向公开世界发布 |

### 2.2 WorldOS 的独特定位

WorldOS 在同类项目中没有直接对标的竞品。它的独特性在于：

1. **世界模型优先**：不是文章列表，而是 Node -> Area -> Relation -> Path -> Timeline 的拓扑结构
2. **权限分层**：公开层 / 私密层 / 创世台三层分离，middleware 服务端守门
3. **门禁驱动**：254 个脚本构成的治理体系，每一轮 RC 都在收紧"什么才算准备好"
4. **AI 边界化**：AI 是灯塔不是太阳，只读公开上下文，有低光降级模式
5. **生命档案**：内容出生证明、世界月报、修复日仪式，让世界可回望可传承

### 2.3 从同类项目学到的教训

| 教训 | 来源 | WorldOS 的应对 |
|---|---|---|
| 数字花园容易变成品展示，缺乏维护节奏 | Maggie Appleton 退役 | 修复日 + 月报仪式 + 低心力维护流程 |
| 静态发布器缺少权限分层 | Quartz / Astro | World Kernel route policy + middleware 服务端拦截 |
| 笔记工具不面向访客 | Logseq / Obsidian | 8 分钟路径 + 反迷路罗盘 + 节点护照标准 |
| AI 集成容易越权 | 通用 AI chatbot | 公开 AI 上下文契约 + 只读公开索引 + 低光降级 |
| 门禁形同虚设则退化为空壳 | RC1 教训 | 内容密度门禁从 30 字符提升到 300 字符 + 契约驱动 |

## 三、全局目标

### 3.1 终极目标（North Star）

**让古月浮屿成为一个可上线、可维护、可传承的个人数字世界产品。**

具体含义：
- 公开世界可被任何人在任何设备上访问、探索和理解
- 私密世界由服务端权限保护，永不泄漏到公开层
- 内容维护可在 1-30 分钟内完成不同档次的动作
- AI 只在灯塔区域出现，只读公开上下文，可降级
- 整个世界可一键导出为 Markdown + JSON + 媒体文件，不依赖任何平台

### 3.2 2026 年度目标

| 目标 | 当前状态 | 年底目标 |
|---|---|---|
| 公开世界上线 | `productionLive: false` | 完成真实外部部署，`productionLive: true` |
| 内容厚度 | 52 节点，平均 500+ 字符 | 80+ 节点，核心节点 2000+ 字符，全部 400+ 字符 |
| 脚本治理 | 254 脚本 | 收束到 200 以内，legacy 脚本归档不动 |
| 移动端体验 | 桌面优先 | 通过移动端验收标准，首屏 < 3s |
| AI 灯塔 | 低光模式可用 | 接入真实 AI，只读公开索引，可降级 |

## 四、里程碑

### M1: 真实上线（K5-external 闭环）
**目标**：完成 `productionLive: true` 所需的全部外部验证
**前置条件**：域名、HTTPS、部署平台
**验收标准**：
- 外部 Preview / Production URL 可访问
- 线上 smoke test 通过
- HTTPS 证书有效
- Web Vitals 快照（LCP < 2.5s, CLS < 0.1）
- 可访问性快照
- 人工签收记录
- 真实回滚演练

### M2: 内容成熟（80+ 节点世界）
**目标**：公开世界从"可走"升级为"值得停留"
**验收标准**：
- 80+ 公开节点，每个 400+ 字符
- 16+ 核心节点，每个 2000+ 字符，3+ 二级标题
- 15+ 公开路径，覆盖所有一级区域
- 120+ 关系星线
- 30+ 世界事件

### M3: 脚本瘦身（200 脚本以内）
**目标**：降低维护负担，让门禁可读可复跑
**验收标准**：
- 总脚本数 <= 200
- 日常入口 <= 10 个命令
- legacy 脚本全部归档到 `scripts/_legacy/`
- 每个保留脚本有用途注释

### M4: 移动端验收
**目标**：掌心里的浮屿
**验收标准**：
- 所有公开路由通过移动端验收标准
- 首屏 LCP < 3s（4G 模拟）
- 无横向溢出
- 单手可达主要入口
- 节点阅读无遮挡

### M5: AI 灯塔点亮
**目标**：AI 从低光模式升级为真实导览
**验收标准**：
- 接入真实 AI provider
- 只读公开世界索引
- 路径推荐可用
- AI 不可用时自动降级为静态索引
- 公开 AI 上下文契约门禁通过

### M6: 传承就绪
**目标**：世界可导出、可回望、可传承
**验收标准**：
- 一键导出 Markdown + JSON + 媒体
- 导出包可在纯静态环境打开
- 内容出生证明覆盖所有节点
- 世界月报连续 6 个月

## 五、阶段规划

### Phase 1: 上线准备（RC9）
**时间**：2-4 周
**主题**：从本地 RC 走向真实部署

| 项 | 说明 | 里程碑 |

| 环境变量与密钥管理 | `.env` 生产配置、API Key 轮换策略 | M1 |
| HTTPS 证书验证 | 部署后验证证书有效性 | M1 |
| 线上 smoke test | 对所有公开路由执行 HTTP 200 + 内容存在性检查 | M1 |
| Web Vitals 快照 | Lighthouse / CrUx 数据采集 | M1 |
| 可访问性快照 | axe-core / Lighthouse Accessibility 评分 | M1 |
| 回滚演练 | 部署后模拟回滚，验证流程可用 | M1 |
| 人工签收 | 签收清单 + 签收记录 | M1 |
| `productionLive: true` | 所有 K5-external 阻断项清零后翻状态 | M1 |

### Phase 2: 内容深耕

| 产品工坊深化 | MoonFrame、AI 小说工作台等种子节点的详细设计文档 | M2 |
| 生活记忆扩展 | 旅行坐标、窗边光等记忆种子的真实内容（脱敏后） | M2 |
| 路径增加 | 从 13 条路径扩展到 15+ 条，覆盖所有一级区域的交叉路径 | M2 |
| 关系星线加密 | 从 82 条关系扩展到 120+，每个节点至少 2 条关系 | M2 |

### Phase 3: 脚本瘦身
**时间**：3-4 周
**主题**：从 254 到 200 以内

| 项 | 说明 | 里程碑 |
|---|---|---|
| legacy 脚本归档 | 将 R1-R8 阶段脚本移动到 `scripts/_legacy/` | M3 |
| 合并同类检查脚本 | 将功能重叠的 check 脚本合并为统一入口 | M3 |
| 删除无用脚本 | 移除从未被 npm scripts 引用的孤儿脚本 | M3 |
| 脚本用途注释 | 每个保留脚本头部添加用途注释 | M3 |
| 命令脊柱文档化 | 更新维护命令脊柱注册表，明确每个命令的用途和适用场景 | M3 |
| CI 精简 | `.github/workflows` 只保留 ci.yml + release-gate.yml | M3 |

### Phase 4: 移动端验收
**时间**：2-3 周
**主题**：掌心里的浮屿

| 项 | 说明 | 里程碑 |

| 首页移动端优化 | 首屏从 5 个入口精简为 3 个，动效降级 | M4 |

### Phase 5: AI 灯塔
**时间**：3-4 周
**主题**：从低光到真实导览

| 项 | 说明 | 里程碑 |
|---|---|---|

| 低光降级 | AI 不可用时退回为静态索引和预设推荐 | M5 |
| 上下文契约门禁 | `check:public-ai-context-contract` 验证 AI 只读公开数据 | M5 |
| 成本控制 | 请求频率限制、缓存策略、降级阈值 | M5 |

## 六、优先级矩阵

| 优先级 | 阶段 | 理由 |
|---|---|---|
| P0 | Phase 1: 上线准备 | 没有真实上线，其他一切都是本地实验 |
| P1 | Phase 2: 内容深耕 | 上线后如果内容薄，世界就是漂亮空壳 |
| P1 | Phase 4: 移动端验收 | 移动端是访客的主要入口，不能等 |
| P2 | Phase 3: 脚本瘦身 | 降低维护负担，但不阻塞上线 |
| P2 | Phase 5: AI 灯塔 | 增强体验，但无 AI 时世界必须完整可用 |
| P3 | Phase 6: 传承就绪 | 长期目标，不急但必须规划 |

## 七、风险与缓解

| 风险 | 概率 | 影响 | 缓解策略 |
|---|---|---|---|
| 域名/部署成本超预算 | 中 | 高 | 优先选 Vercel 免费额或 Cloudflare Pages 免费额 |
| AI provider 涨价或停服 | 中 | 中 | 低光降级模式已就绪，世界不依赖 AI |
| 内容维护疲劳 | 高 | 高 | 低心力维护流程 + 修复日节奏 |
| legacy 脚本回流 | 低 | 中 | 导入边界门禁 + mainline governance 检查 |
| 私密内容泄漏 | 低 | 极高 | middleware 服务端拦截 + 公开构建只读 public 数据 |
| 移动端性能不达标 | 中 | 中 | 性能预算 + 动效降级 + 构建产物监控 |

## 八、不做清单（Anti-Goals）

以下事项在当前阶段明确不做：

- 不新增 R8.x / V / R 阶段线，不再造动态宇宙 runtime
- 不把私密权限交给前端硬编码
- 不让 build wrapper 代替真实生产证据
- 不让世界语言牺牲现实可理解性
- 不为 AI 牺牲无 AI 可用性
- 不把门禁做成摆设
- 不在未上线前过度工程化

## 十一、技术演进方向

### 11.1 短期（Phase 1-3）

- 保持 Next.js 15 + React 19 + Tailwind 技术栈稳定
- 构建产物监控，防止旧 runtime 回流
- 脚本从 254 收束到 200 以内
- 内容密度门禁从 300 提升到 400 字符

### 11.2 中期（Phase 4-5）

- 移动端从桌面优先改为响应式优先
- AI 灯塔接入真实 provider，保持低光降级
- 性能预算从口号变为门禁
- 搜索体验从 Fuse.js 升级为全文检索

### 11.3 长期（Phase 6+）

- 导出管线从手动变为一键
- 内容出生证明全覆盖
- 世界协议文档化，让其他 AI 工具可读可审计
- 探索联邦化可能：多个个人世界之间的星线互联

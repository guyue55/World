# WorldOS 真格世界实现差距分析

> [!IMPORTANT]
> 本文档基于 2026-07-10 真实视觉审查，重新定义“现有距离真正世界 / 宇宙体验还差什么”。它用于修正此前“底座完成 ≠ 体验完成”的口径。

## 1. 一句话判断

WorldOS 当前不是空白，也不是白屏；但距离高目标仍差一个核心层级：

> 现有是“页面舞台化的开端”，目标是“场景主体可运行的世界”。

## 2. 层级差异

| 层级 | 当前是否具备 | 用户感知 | 目标状态 |
| --- | --- | --- | --- |
| 静态内容 | 基本具备 | 能读 | 能作为世界事实源被多场景吸收 |
| 动态背景 | 具备 | 有一点动效 | 动效服务场景叙事 |
| 场景壳 | 首轮具备 | 有舞台感 | 不只是壳，场景主体可操作 |
| 场景人格 | 部分具备 | 文案区分大于体验区分 | 一眼可区分空间、交互和节奏 |
| 场景迁移 | 部分具备 | 更像提示条 | 有来源、离开、预告、抵达、沉淀 |
| 内容节点 | 部分具备 | 仍像文章 | 像地点，有护照、关系门和出口 |
| 灯塔 AI | 边界具备 | 体验弱 | 只读导览、解释、推荐、回退 |
| 声景 / 氛围 | 基础具备 | 不遮挡 | opt-in、轻量、场景化、有授权 |
| 质量证据 | 基础具备 | 以前容易自证 | 截图 / 录屏 / 人工量表共同判定 |

## 3. 根因

### 3.1 把“统一组件”误当成“统一世界”

统一 `SceneWorldPortal` 能降低开发成本，但如果所有场景共用同一种布局和节奏，用户看到的是同一张模板换文案。

后续原则：

- 统一数据契约。
- 统一运行时状态。
- 统一质量门禁。
- 不统一场景主体交互。

### 3.2 把“通过检查”误当成“体验达标”

`check:mainline`、`release:local-rc`、脚本报告只能证明没有明显工程故障。它们不能证明页面不再像骨架。

后续必须加入：

- 逐页截图审查。
- 关键路径录屏。
- 场景主体可见性检查。
- 人工体验量表。
- “仍像骨架则不能完成”的否决项。

### 3.3 把“舞台背景”误当成“场景主体”

Atlas 的主体不是星图背景，而是可探索关系地图。Timeline 的主体不是曲线，而是事件水位和回看路径。Archive 的主体不是书架线条，而是检索和归档空间。Paths 的主体不是虚线，而是可走的旅程。

## 4. 场景级缺口

| 场景 | 当前最像骨架的地方 | 真格目标 | 关键实现方向 |
| --- | --- | --- | --- |
| Home | 首访仍像介绍 | 进入世界仪式 | 抵达状态、世界选择、入口分流、首次 / 再次访问差异 |
| Atlas | 右侧星图不可探索 | 世界地图 | 区域聚焦、节点层级、关系解释、回路径 |
| Timeline | 时间河不承载事件 | 时间流 | 事件河道、年份 / 阶段锚点、节点回看 |
| Archive | 下方还是密集列表 | 档案馆 | 检索大厅、分区、筛选、卷宗详情 |
| Paths | 路线不具行走感 | 旅程系统 | 进度、下一站、完成态、返回地图 |
| Node | 文章详情 | 地点房间 | 节点护照、关系门、下一站、所在区域 |
| Transition | 页面提示 | 场景迁移 | 来源残影、目标预告、抵达沉淀 |
| Lighthouse | 问答页 | 观测站 | 只读事实源、导览建议、边界解释、失败回退 |

## 5. 开发策略修正

### 5.1 不再继续堆总控文档

现有总控文档已经够多。后续新增文档只允许服务三件事：

- 修正完成定义。
- 明确场景主体。
- 约束验收证据。

### 5.2 不再继续加统一壳

后续开发要从“共享壳”转为“场景主体组件”：

| 场景 | 应新增 / 强化的主体组件 |
| --- | --- |
| Home | `WorldGatewayStage` |
| Atlas | `AtlasExplorationStage` |
| Timeline | `TimelineRiverStage` |
| Archive | `ArchiveHallStage` |
| Paths | `JourneyRouteStage` |
| Node | `NodePlaceRoom` |
| Lighthouse | `LighthouseGuideStage` |
| Transition | `SceneMigrationOverlay` |

### 5.3 不用重型 3D 作为捷径

高目标不等于上来引入全站 3D。当前更适合：

- SVG / Canvas 作为可视化主体。
- GSAP 做场景编舞。
- CSS 和原生 Web API 做轻量氛围。
- Web Audio 只做用户主动开启的短声景。
- 3D / WebGL 只作为后续局部增强候选，不作为主线依赖。

## 6. 进入开发前的新门槛

每个阶段开始前必须回答：

1. 这个阶段修的是哪个真实截图里暴露的问题？
2. 这个页面的“场景主体”是什么？
3. 用户第一眼或第一交互内能看到什么变化？
4. 静态降级后还能读到什么？
5. reduced-motion 后还能理解什么？
6. 这次是否只是换文案 / 换背景？
7. 截图和录屏如何证明它不像骨架？

若答不清，不能进入开发。

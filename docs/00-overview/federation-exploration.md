# 古月浮屿联邦化探索

> 日期：2026-07-08
> 状态：概念草案
> 前置文档：world-protocol-specification.md

## 一、愿景

联邦化是古月浮屿的长期愿景：多个个人世界之间通过协议互联，形成去中心化的世界网络。每个世界独立运行，通过星线跨世界引用彼此的公开节点。

这不是社交网络，也不是内容平台。它是个人世界之间的友邻链接--基于协议、基于信任、基于自愿。

## 二、互联模型

### 2.1 点对点

联邦化不依赖中心服务。每个世界独立部署，通过 HTTP 获取对方世界的公开索引。

### 2.2 星线跨世界

当前的关系星线只在同一世界内连接节点。联邦化后，星线可以跨世界：

- 世界 A 的 AI 边界节点 -> 世界 B 的隐私保护节点
- 跨世界关系需要双方世界都公开了索引

### 2.3 信任目录

每个世界维护一份已知世界列表，类似友邻链接：

```json
{
  "knownWorlds": [
    {
      "url": "https://friend.example.com",
      "name": "友邻世界",
      "addedAt": "2026-07-08",
      "trust": "bidirectional"
    }
  ]
}
```

信任关系可以是单向的（我引用你）或双向的（互相引用）。

## 三、发现机制

### 3.1 手动添加

世界维护者手动在信任目录中添加友邻世界的 URL。

### 3.2 间接发现

通过已有友邻的信任目录，间接发现新的世界（友邻的友邻）。

### 3.3 目录服务（未来）

未来可能有社区维护的世界目录服务，但不作为核心依赖。

## 四、技术挑战

### 4.1 信任模型

如何确认对方世界的身份？初步方案：URL 本身就是身份，类似于友邻链接。更严格的方案可引入签名机制。

### 4.2 数据同步

跨世界关系如何保持一致？初步方案：关系存储在发起方世界，对方世界不需要感知。这类似于 Web 链接--你链接到别人的页面，但对方的页面不需要知道。

### 4.3 权限控制

联邦化只涉及 public 可见性的节点。私密层永远不进入联邦。

### 4.4 内容漂移

对方世界的节点可能被修改或删除。初步方案：缓存对方索引快照，定期刷新，删除时显示远方节点已不可达。

## 五、实现路径

### Phase 1：索引暴露（当前）

- public/world-index.json 已存在
- 可通过 HTTP 访问

### Phase 2：信任目录

- 创建 known-worlds.json
- 在 Atlas 中展示远方世界区域

### Phase 3：跨世界星线

- relations.json 支持 crossWorld 类型
- Atlas 可视化跨世界星线

### Phase 4：联邦搜索

- 灯塔搜索可搜索友邻世界的公开节点
- 结果标注来源世界

## 六、当前状态

联邦化是概念草案阶段。world-index.json 已具备类似结构，但尚未暴露为标准端点。信任目录和跨世界星线尚未实现。

本探索文档记录方向和初步设计，为未来实现提供参考。不设时间表，不作为当前开发优先级。

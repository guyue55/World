# WorldOS Reality-First 总控契约

> [!IMPORTANT]
> 这是 Reality-First 单次 Goal 的最高项目契约。它取代旧 M8-M30 的完成口径。Goal 执行期间不得由执行者修改本文件来降低目标或绕过验收。

## 1. 结论先行

WorldOS 当前具备大量工程底座，但真实产品状态仍是：

> **具有动态背景、统一舞台壳和工程门禁的博客式骨架。真实世界 / 宇宙体验尚未完成。**

旧 M30 的 `8.9`、`9/10 候选`与“八支柱完成”结论作废，原因包括：

- 终局脚本直接硬编码 S1-S8 的体验分数。
- `candidateScore` 使用最低阈值托底，不能反映页面质量。
- 视觉检查主要验证截图存在、文件大小、DOM token、无遮挡和无横向溢出。
- 已归档录屏仍显示统一双栏 hero、状态卡、工程说明和列表式内容。
- 执行者同时定义标准、实现功能、生成报告并批准自己，缺少独立视觉否决机制。

Reality-First 不延续旧编号，也不再通过增加“下一轮”移动终点。它是一个目标、一次触发、十个内部检查点和一个固定停止条件。

## 2. 需求契约

- **核心场景**：访问者在 localhost 或 LAN IP 打开古月浮屿，第一眼明确感到进入一个个人世界，并能在不同空间中探索、迁移、阅读、问路和继续上次旅程。
- **用户价值**：降低理解门槛；把内容、项目、记忆与规则组织为可感知空间；形成愿意停留和回访的个人数字世界，而不是项目说明站。
- **主要工作**：重做世界入口；为 Atlas、Timeline、Archive、Paths、Node、Lighthouse 建立独立场景主体；重做迁移、声景和导览；清理公开工程文案；建立真实视觉验收闭环。
- **事实源**：公开世界对象继续来自现有 `data/domains/experience`、`data/core`、`content` 与服务端 public index；权限事实来自服务端 / 数据过滤契约；运行时状态来自 World Runtime；验收事实来自新鲜构建、浏览器、截图、录屏和独立审查。
- **不做的**：外部 Preview / Production、域名、HTTPS、多人在线、全站 3D、WebXR、社交网络、无必要数据库迁移、为了效果重写整个 World Kernel。
- **异常流**：无 JavaScript、无存储、无音频权限、减少动效、窄屏、AI Provider 不可用、内容为空、API 错误和无权限时，都必须有可理解且不泄漏的路径。
- **成本限制**：复用现有栈；视觉资产和音频受预算约束；新依赖需 ADR；不以 200 个节点全部首屏渲染；不为历史脚本继续扩张主线。
- **协作角色**：Goal 执行者负责实现；Reality Auditor 或独立审查上下文负责验收；用户只负责触发 Goal，不承担逐阶段确认。
- **优先级**：世界体验重构为 P0；稳定、权限、无障碍和性能为并行 P0；实时 AI Provider 与跨设备同步为条件性 P1；全站 3D 为 P3。

## 3. 唯一目标

把当前公开主线重构为一个可运行的本地个人数字世界，使用户能完成以下连续旅程：

```text
抵达入口
  -> 选择方向
  -> 在 Atlas 中定位一个区域或节点
  -> 穿梭到 Timeline / Archive / Paths
  -> 抵达 Node 地点并阅读
  -> 通过关系门或灯塔继续探索
  -> 再次访问时从上次位置继续
```

这个旅程必须由真实场景、交互和内容承担，不能依靠页面上的解释文字告诉用户“这里是世界”。

## 4. 固定停止条件

只有同时满足以下条件，Goal 才能结束：

1. 执行计划 C0-C9 的全部复选框为 `[x]`，且每项有提交、命令和证据。
2. Home、Atlas、Timeline、Archive、Paths、Node、Lighthouse 七类空间均通过体验验收。
3. 隐藏主要文字后，七类空间仍能凭构图、空间对象和交互被区分。
4. 首访、场景迁移、Atlas 探索、Timeline 回看、Archive 检索、Path 行走、Node 阅读、Lighthouse 导览、回访继续九条流程均有连续录屏。
5. desktop、mobile、reduced-motion、reduced-sensory 均完成真实浏览器验收。
6. 公开页面不再出现工程验收文案、统一 `SceneWorldPortal` hero 或可见 QA 说明。
7. 无白屏、遮挡、溢出、死链、焦点丢失、默认声音、权限泄漏、私密数据进入公开载荷、前端 AI Key。
8. 性能、资源、无障碍、权限和 AI 边界达到冻结预算；本地作者 preview / apply / rollback 在隔离 workspace 中真实演练通过。
9. `typecheck`、`lint`、真实生产构建、主线检查和本地 / LAN RC 通过，且证据时间晚于最后源码改动与构建。
10. 独立 Reality Audit 对每项只给 `pass / fail / blocked`，所有 P0/P1 为零，所有核心体验为 `pass`。

以下均不是停止条件：阶段编号完成、脚本输出 passed、截图文件存在、没有白屏、某个页面“有动效”、某份报告自报 9 分。

## 5. 成熟度口径

为避免再次夸大，只允许以下四种状态：

| 状态 | 含义 | 能否完成 Goal |
| --- | --- | --- |
| `FOUNDATION_ONLY` | 工程底座可运行，体验仍像骨架 | 否 |
| `WORLD_EXPERIENCE_IN_PROGRESS` | 部分场景已重做，连续旅程未闭环 | 否 |
| `LOCAL_WORLD_COMPLETE_AI_FALLBACK` | 本地 / LAN 世界体验全部通过，灯塔以公开事实源低光模式运行 | 是，但必须明确无实时生成式 Provider |
| `LOCAL_WORLD_COMPLETE_LIVE_AI` | 上述全部通过，且实时服务端 Provider 也通过成本、权限、grounding 和回退验收 | 是 |

不得再使用无来源的 `8.9`、`9/10`、`终局`、`完美`或“彻底完成宇宙”等数字和绝对表述。

## 6. 范围

### 6.1 包含

- `/` 世界入口和首访 / 回访差异。
- `/atlas` 空间地图与节点关系探索。
- `/timeline` 时间河、阶段锚点和事件回看。
- `/archive` 检索大厅、分区、筛选和卷宗入口。
- `/paths` 与 `/paths/[id]` 的路线选择、行走和完成感。
- `/node/[slug]` 的地点护照、正文、关系门和出口。
- `/ask` 的灯塔空间、问路、依据、推荐和失败回退。
- 全局 shell、导航、迁移、世界状态、回访记忆、声景和降级。
- `/status` 作为唯一工程证据面板，但不进入访客主旅程。
- 公开数据、AI 上下文和权限边界回归检查。
- 一个本机 CLI 作者入口，用中文 draft 完成影响预览、原子写入和可验证回滚；不在 LAN 暴露写 API。

### 6.2 排除

- 外部网络发布、云服务、域名和生产上线。
- 自动生成或伪造用户的真实人生经历、照片、家人信息或私密档案。
- 没有用户授权的第三方音乐、图片、字体、模型或纹理。
- 需要持续 GPU 高占用的全站 3D 世界。
- 多人协作、账号体系、跨设备同步和消息系统。

### 6.3 静态 / 动态双形态

WorldOS 不是两套网站，而是同一事实源的两种体验层：

| 形态 | 最终目标 | 验收 |
| --- | --- | --- |
| 静态形态 | Server-rendered 内容、地图等价列表、时间锚、档案检索入口、路径顺序、节点正文和低光导览在 JavaScript 失败 / 关闭时仍可读可走 | JS-off route smoke、链接、正文、权限载荷、无白屏 |
| 动态形态 | 在静态内容上渐进增强为空间地图、时间河、档案大厅、旅程、地点、迁移、回访和 opt-in 声景 | 场景交互、截图、录屏、性能和 reduced-motion |

两种形态必须共享 public index、权限过滤、内容 schema 和 URL。禁止维护一套静态博客和另一套动态宇宙，导致内容、权限或导航分叉。

## 7. 产品方向

### 7.1 世界观

统一视觉母题为：

> **月下浮屿中的个人观测与记忆系统。**

它不是紫色霓虹星空模板，也不是写实电子游戏。世界由浮岛、星图、河流、档案、道路、房间和灯塔构成；真实内容、关系和时间是这些空间的材料。

### 7.2 入口原则

- 首屏首先展示可操作世界，不先展示产品说明。
- 品牌名是第一视口信号，但不占据大半屏幕。
- 首访只提供一个明确动作和少量方向；回访优先显示“继续上次旅程”。
- 首屏必须露出下一层内容或可探索对象，避免封闭的 hero 卡片。

### 7.3 场景原则

- Atlas 的主体是可探索地图，不是地图介绍。
- Timeline 的主体是承载事件的河道，不是事件列表上方的一条曲线。
- Archive 的主体是检索与分区空间，不是换了背景的卡片列表。
- Paths 的主体是路线和站点，不是路径详情加侧栏进度。
- Node 的主体是内容地点和关系出口，不是文章详情旁边加护照卡。
- Lighthouse 的主体是陪伴式导览，不是问答面板放在灯塔插画下面。

## 8. 权威事实与禁止自证

| 判断 | 权威事实 | 弱证据 / 禁止替代 |
| --- | --- | --- |
| 页面是否可用 | 生产态浏览器真实交互 | 源码包含某字符串 |
| 是否像独立空间 | 新鲜截图、隐藏文字截图、连续录屏、独立审查 | variant 数量、组件数量 |
| 是否完成迁移 | 实际导航录屏、状态与焦点检查 | transition registry 存在 |
| 权限是否可靠 | 服务端过滤、响应载荷、公开产物扫描 | 前端隐藏按钮 |
| AI 是否真实 | Provider 响应元数据、来源、模式标记、审计 | 配置文件写 enabled |
| 性能是否合格 | 最新生产构建、浏览器指标、资源清单 | 依赖数量未增加 |
| 证据是否新鲜 | source -> build -> server -> evidence 的时间链 | 旧报告 status=passed |

执行者不得：

- 为体验维度写固定高分。
- 以最低阈值托底计算结果。
- 让检查脚本只验证自己刚生成的报告状态。
- 把同一执行上下文的主观自评称为“人工审查”。
- 为了通过门禁修改冻结验收标准。

## 9. 长线执行协议

Goal 每次启动、续跑或上下文恢复必须：

1. 读取 `AGENTS.md` 和六份权威文档。
2. 读取 `git status --short --branch`，保护现有改动。
3. 从执行账本找到第一个未完成检查点。
4. 检查上一个提交、失败日志和证据新鲜度。
5. 只完成当前检查点的最小完整闭环。
6. 运行定向检查并真实打开页面。
7. 更新计划复选框和执行账本。
8. 中文提交。
9. 自动进入下一个检查点，不等待用户重复指令。

任何单个检查点完成都不是 Goal 完成。只有 C9 终局审查满足第 4 节，才可结束。

## 10. 失败与恢复

- 命令失败：记录命令、退出码和关键错误；定位根因；最小修复；重跑失败项及受影响主线。
- 页面异常：保存 before / failure 截图；修复；重新构建并从生产服务器复查。
- 证据过期：删除或标记 stale，重新构建、启动、截图和录屏。
- 端口占用：检测并选择空闲端口，不杀死未知用户进程。
- 浏览器不可用：优先系统 Chrome；再使用已安装 Playwright CLI；记录替代路径。
- 上下文压缩：只从执行账本恢复，不重新发明路线，也不重做已提交且证据有效的检查点。
- 外部秘密缺失：继续完成所有非秘密工作；实时 AI 标记为条件性阻塞，不得伪造；本地低光模式可按第 5 节完成。

## 11. 外部前置条件的诚实边界

当前仓库未检测到 `OPENAI_API_KEY`，也未检测到本地 Ollama。任何 Goal 提示词都不能凭空创造真实 Provider 凭据。

因此：

- 世界体验、场景、迁移、内容、声景、权限、性能和低光灯塔可以自动完成。
- Goal 可以实现服务端 Provider 适配、schema、限流、超时、审计、成本统计和 fallback。
- 只有运行环境确实提供合法 Key 或受控本地模型时，才可验证 `LOCAL_WORLD_COMPLETE_LIVE_AI`。
- 未提供时，最终允许的诚实状态是 `LOCAL_WORLD_COMPLETE_AI_FALLBACK`，不是“实时 AI 已完成”。

## 12. 授权边界

用户触发本 Goal 即表示：已批准本控制包内的本地代码、内容结构、视觉资产、测试、报告和 Git 提交工作；无需每个检查点重复请求批准。

该授权不包括：

- 外部部署或对公网开放服务。
- 购买付费资产或服务。
- 创建、读取或上传用户未提供的秘密。
- 删除用户资料、重写 Git 历史或回滚未知改动。
- 公开私密内容。

遇到这些边界时，完成所有可控工作并在账本中诚实记录，不擅自扩大权限。

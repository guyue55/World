# 古月浮屿世界宣言

一张书桌连接星河。

这里不是一个完成的网站，而是一个正在生长的个人数字世界。

我把技术、产品、灵感、生活与记忆一点点安放在这里，让它们在时间中自然生长。

## 为什么不是普通博客

普通博客通常按时间倒序堆放文章，读者需要自己判断从哪里开始、哪些内容相关、哪些只是历史痕迹。古月浮屿想解决的不是“多发几篇文章”，而是把内容放回一个可以进入、可以返回、可以持续维护的世界。

在这里，一篇技术记录可以是一颗星，一个生活片刻也可以是一颗星。地图负责说明它属于哪片区域，路径负责告诉第一次来的人先看什么，档案馆负责让熟悉关键词的人快速检索，时间线负责保留世界生长的痕迹。这样，内容不只是被发布，而是被安放。

## 公开层的承诺

公开层只展示已经允许公开、可以被陌生访问者理解的内容。它不会假装自己就是完整世界，也不会把私密、家庭、伴侣、保险箱或封存内容混进前台索引。访问者看到的是外庭，是入口，是可以安全走动的部分。

这条边界很重要。世界要有深度，但入口必须清澈；系统要有能力，但不能让能力越过权限；AI 可以帮忙照路，但不能替造物主决定道路。古月浮屿的第一条产品原则，就是让第一次来的人不需要懂项目历史，也能知道自己在哪里、能去哪、如何回来。

## 如何继续生长

世界会以低频、可维护的方式继续生长：补充真实节点，整理关系星线，更新路径，复核公开边界。每一次增长都应该让世界更好逛，而不是只让文件更多。一个新节点如果没有摘要、没有区域、没有关系、没有路径，它就还没有真正进入世界。

所以这份宣言不是终点，而是一张入场券。你可以先走新手路径，看见它的骨架；也可以打开地图，从区域进入；或者去档案馆检索现实问题。只要还能回来，只要边界清楚，只要内容继续被照料，这片星河就还在生长。


## 世界的四个面向

古月浮屿同时面向四个方向，每个方向都有明确的产品要求：

- **对外**：可进入、可探索、可停留的公开世界。访客不需要注册、不需要登录、不需要理解项目历史就能逛懂。
- **对内**：可维护、可记录、可整理的创世台。维护动作分为 1 分钟、5 分钟、15 分钟和 30 分钟四档，避免世界成为负担。
- **对未来**：可导出、可回望、可传承的生命档案。所有内容以 Markdown 和 JSON 保存，不依赖任何平台或数据库。
- **对 AI**：可读、可审计、可边界化协作的世界协议。AI 只读公开上下文，建议进入审核队列，不直接修改世界。

这四个面向不是功能列表，而是约束条件。每一个新功能、每一个新节点、每一次维护动作都应该同时通过这四个方向的检验。如果一个改动让对外更难进入、让对内更难维护、让未来更难导出、让 AI 更难审计，那它就需要重新考虑。

## 边界即自由

很多人觉得边界是限制，但在个人数字世界里，边界恰恰是自由的前提。没有权限边界，私密内容迟早会泄漏到公开层；没有内容密度门禁，世界会退化为空壳；没有脚本治理，维护成本会指数增长；没有 AI 边界，灯塔会变成越权引擎。

古月浮屿的边界体系包括：路由边界（middleware 服务端拦截）、API 边界（注册表 + 权限校验）、内容边界（密度门禁 + 契约驱动）、脚本边界（legacy 注册表 + 命令脊柱）和 AI 边界（公开上下文契约 + 低光降级）。这些边界不是一次性的防护墙，而是每次开发时都会跑的门禁——它们让"不小心犯错"变得困难。

## 这份宣言的承诺

这份宣言对访客承诺：你看到的世界是真实的，不是营销页面；你可以随时找到回去的路；你的隐私不会被收集；你不需要懂技术就能逛。对造物主承诺：世界可以在低心力下维护；内容不会因为平台关闭而消失；AI 不会替你做决定；每一次维护都有门禁守护。对未来的自己承诺：所有内容都有出生证明；世界可以被一键导出；月报和修复日记录了世界如何生长；传承协议告诉继承者怎么处理这一切。

## 访客的旅程

一个访客第一次来到古月浮屿时，他的旅程应该是这样的：首页告诉他"这里是什么"和"从哪开始"；8 分钟路径给他一条完整但简短的主线；地图让他看见世界的空间分布；时间线让他感受世界在生长；档案馆让他在需要时快速找到具体内容；灯塔在他迷路时提供低光导览。

这条旅程的核心是"不迷路"。世界设计公理说"入口清澈，深处浩瀚"——入口做减法，只展示最少必要信息；深处做加法，愿意深入的访客能发现关系星线、时间线回望和区域关联。中间用路径引导，确保从概览到深读的过渡是平滑的。

如果访客在任何时刻不知道"我在哪、能去哪、怎么回去"，那就是世界的导航失败，需要修复。这也是为什么每个公开页面都有 `ProductRouteGuide` 组件——它告诉访客当前位置、现实解释和下一步选项。

## 世界的维护节奏

世界不是一次性建成的，它需要持续维护。维护节奏分为四档：1 分钟记一粒光（灵感碎片），5 分钟整理一颗种子（给节点补摘要或标签），15 分钟连接几颗星（补充关系星线），30 分钟修复一个小区域（批量补内容或修格式）。

维护不需要每天做。当世界积累了足够多的未整理内容，或者某个区域明显变薄时，就触发一次修复日。修复日不新增概念，只做清尘：整理灵感、修复关系、补摘要、检查权限、归档沉睡项目。修复完成后记入世界日志，让维护本身也成为时间线的一部分。

每月做一次月报，回望新增节点、活跃主题、沉睡灵感和下一步。每季做一次季报，检查方向是否偏离初衷。每年做一次年度回望，看世界如何生长、哪些种子发芽、哪些沉睡。这种节奏让世界不会因为缺乏维护而荒废，也不会因为过度维护而成为负担。

## 世界的公开边界

公开世界不是完整世界。访客看到的是经过筛选、脱敏和整理的公开层——大约是完整世界的三分之一。剩下的三分之二是私密层（家庭、伴侣、保险箱）和创世台（草案、审计、治理）。

这个比例不是随意的。公开层需要足够丰富，让访客觉得"这里值得停留"；但不能太完整，否则私密边界会模糊。当前 52 个公开节点、13 条路径、82 条关系和 21 个世界事件构成了公开层的骨架。未来目标是 80+ 节点和 120+ 关系，让骨架变得更丰满。

公开边界的守门人是 middleware。它在服务端拦截所有 private 和 internal 路由，重定向到 /forbidden 或 /archive。前端组件只负责展示已经过滤的数据，不承担"能不能看"的判断。这保证了即使有人直接输入私密路由的 URL，也无法绕过权限。

## 诚实是第一产品力

古月浮屿的工程状态段落始终如实记录当前进展：`productionLive: false` 意味着"还没上线"，不伪装已上线；内容密度门禁 300 字符意味着"每个公开节点至少有一段真实正文"，不接受只有标题的空壳；脚本 legacy 注册表 254 条意味着"有 254 个脚本需要治理"，不假装脚本已经收束。

诚实不等于保守。世界在持续生长——内容在补厚，移动端在适配，AI 灯塔在规划。但每一个进步都有门禁验证、有证据存档、有 git 记录。这种"用证据说话"的文化，是古月浮屿作为生命档案的根基。

## 世界的边界是诚实的延伸

诚实不只是"不撒谎"，还包括"知道自己的边界在哪"。古月浮屿明确不做的事情比做的事情更重要：不新增阶段线、不把私密权限交给前端硬编码、不让 build wrapper 代替真实生产证据、不为 AI 牺牲无 AI 可用性。这些"不做"让世界保持聚焦，避免在收束期重新膨胀。

边界的另一个含义是权限分层。公开层、私密层和创世台三层分离，由 middleware 在服务端守门。前端只控制体现，不控制权限——私密内容不会因为前端 CSS 隐藏就安全，只有服务端拦截才是真正的边界。这种"后端控制权限，前端控制体现"的设计，让世界在任何客户端环境下都保持一致的隐私保护。

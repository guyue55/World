# Gemini CLI、Skills 与 Agent 工作流

这个节点记录把 CLI、Skills 和外部工具接入个人工作流的经验。

它的重点不是单一模型，而是工程化调度：

- 先明确任务边界，再选择工具。
- 让技能承担可复用步骤，而不是临时提示词堆叠。
- 对飞书、GitHub、Google Workspace 等连接器保持权限最小化。
- 所有 AI 生成建议进入草案区，不直接覆盖原始记录。

这条经验会回流到 AI 灯塔和创世台：AI 可以帮忙整理，但不能替创作者决定世界法则。

## 从提示词到工作流

单次提示词解决的是眼前任务，工作流解决的是可重复性。Gemini CLI、Skills、Codex、Playwright、各种连接器真正有价值的地方，不是“某个模型回答得好”，而是把一组可靠步骤固定下来：读上下文，形成计划，执行小改动，跑检查，保留证据，提交可复核的变更。

这套流程对 WorldOS 尤其重要。因为世界不是一次性的页面，而是长期维护的系统。没有工作流，项目会很快堆满临时脚本、临时页面、临时想法；有了工作流，每一次改动都能被问清楚：它属于哪个阶段，解决哪个问题，是否影响权限，是否通过本地 RC。

## Skills 的位置

Skills 不应该变成神秘咒语，而应该像工程手册。写计划时使用计划技能，调试时使用系统化调试，完成分支前使用验证和收尾流程。这样做不是仪式感，而是降低遗漏概率。尤其在多轮开发里，技能能让上下文压缩之后仍然保留操作纪律。

同时，Skills 也必须服从项目风格。它们不能鼓励绕过本地检查，不能替代真实运行结果，更不能把外部部署当作当前目标。当前阶段的核心是本地成熟度，所以所有工作流都要回到 localhost / LAN IP、内容质量、动态体验和权限边界。

## 对 AI 灯塔的影响

未来灯塔如果接入真实 Provider，也应该沿用这套工作流思想。它不能直接写文件，而是生成建议；不能自己发布，而是进入审核；不能读取全部上下文，而是由后端裁剪公开或授权数据。AI 的能力越强，流程越要清楚。

这个节点因此不只是技术笔记，也是世界治理原则的一部分：工具可以变，模型可以换，但“可复跑、可审计、可回滚、可解释”的工作流必须留下来。


## 工作流的六个步骤

从 CLI 工具到 Skills 再到 Agent 工作流，最终沉淀为六个可重复的步骤：

1. **读上下文**：先运行 `repo_map.py` 或 `rg` 了解项目结构，不做盲改。
2. **形成计划**：用计划技能把任务拆解为可执行步骤，每步有验收标准。
3. **执行小改动**：每次只改一个模块，改完跑 `check:daily`。
4. **跑检查**：lint、typecheck、content check、boundary check 逐项通过。
5. **保留证据**：门禁输出、构建产物、smoke 报告存档到 `docs/90-archive/reports/`。
6. **提交可复核变更**：Git commit 格式 `xxx(xxx): 中文xxx`，中文注释说明做了什么。

这六个步骤不是理论框架，而是从大量实践中总结出来的。跳过任何一个步骤都会导致问题：不读上下文会改错地方；不形成计划会做超出范围的事；不跑小改动会导致大范围回滚；不跑检查会引入故障；不保留证据无法回溯；不按格式提交会让历史记录混乱。

## Skills 的维护原则

Skills 是工程手册，不是神秘咒语。一个好的 Skill 应满足三个条件：可读（任何人打开 SKILL.md 能理解它在做什么）、可跑（有明确的执行步骤和验收标准）、可审计（执行过程留下可追溯的证据）。

Skills 不应该鼓励绕过门禁。如果一个 Skill 说"直接提交不用跑检查"，它就违反了项目守则。Skills 也不能替代真实运行结果——"应该能通过"不等于"通过了"。当前阶段所有工作流都要回到 localhost / LAN IP、内容质量、动态体验和权限边界。

## 多工具协作的现实

现实中不会只用一个工具。Codex 擅长代码编辑，Claude Code 擅长长文本分析，Gemini CLI 擅长多模态理解。多工具协作的关键不是"哪个最强"，而是统一工作流：不管用哪个工具，都走同样的六步流程，都遵守同样的门禁，都留下同样格式的证据。

这样即使工具换了，工作流不变，世界仍然可维护。这也是为什么世界设计公理说"规则是世界的自然法则"——工具是会换的，规则不应该跟着换。

## 工作流的失败模式

多工具协作最常见的失败模式有三种。第一种是"上下文丢失"：用工具 A 做了一半，切换到工具 B 时上下文被压缩，B 不知道 A 做了什么。解决方式是把关键决策和已完成步骤写入文件（计划文档、git commit），而不是依赖对话上下文。

第二种是"门禁绕过"：某个工具说"看起来没问题"，开发者就跳过了 `check:daily`。但"看起来没问题"不等于门禁通过。解决方式是无论哪个工具做的改动，都必须跑完整的门禁链。门禁是客观的，工具的判断是主观的。

第三种是"范围蔓延"：原本只改一个文件，工具建议顺便改其他文件，结果改了十个文件引入三个故障。解决方式是每次只做一个任务，改完就提交。如果工具建议改其他东西，记录到待办列表，不在当前任务中做。

## Skills 的演进

Skills 不是一次性写好的。它们需要在使用中不断打磨——发现遗漏的步骤就补充，发现不必要的步骤就删减，发现可以自动化的步骤就脚本化。好的 Skills 像好的代码一样，经过多次迭代后变得简洁可靠。

WorldOS 项目本身也在用 Skills 驱动开发。比如编写计划时使用 writing-plans 技能，调试时使用 systematic-debugging 技能，完成分支前使用 verification-before-completion 技能。这些技能不是装饰，而是降低遗漏概率的工程手段。它们让上下文压缩后仍然保留操作纪律，这对于长期维护的项目至关重要。

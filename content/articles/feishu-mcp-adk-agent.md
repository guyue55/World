# 飞书 MCP 接入 ADK Agent

这是一个技术探索节点，用于记录飞书 MCP、ADK Agent、CLI 工具和 Skills 机制之间的连接方式。

在古月浮屿中，它属于技术星域，也被纳入“技术与 AI 探索路径”。

## 这类节点为什么进入公开世界

飞书 MCP、ADK Agent、CLI 和 Skills 看起来是工具细节，但它们反映的是一个更大的问题：个人工作流如何从“手动复制粘贴”走向“可审计的工具编排”。公开记录这些探索，不是为了展示工具名单，而是为了保留方法：如何判断一个连接器是否值得接入，如何限制权限，如何让 AI 只做建议而不是直接执行。

在世界结构中，这个节点连接技术星域和灯塔边界。它说明 AI 能力真正落地时，必须经过工具协议、权限模型和审核队列，而不是靠一句“请帮我处理所有事情”的提示词。

## 接入时优先看的边界

第一是身份边界。连接器能代表谁访问数据，必须由服务端或授权层决定，不能在前端写死。第二是动作边界。读取、搜索、草拟和提交是不同级别的动作，不能因为都叫“自动化”就混在一起。第三是证据边界。每一次工具执行都应该能留下日志、报告或审核状态，方便以后追溯。

这些边界和 WorldOS 的权限准则是一致的：后端控制权限，前端只体现结果。飞书这类协作系统尤其需要谨慎，因为它连接的往往是真实组织、真实消息和真实文档。任何便利都不能越过最小权限原则。

## 对本地世界的启发

当前阶段仍以本地 LAN 为主，所以这类能力不会急着接入真实生产流程。但它已经能指导 Owner 工作台的设计：建议可以进入队列，维护动作可以生成草案，导出可以有任务记录，审核必须由人确认。

当这个节点被放入技术路径，读者看到的不只是一个工具案例，而是一条从“工具很酷”走向“工具可控”的路线。成熟的本地世界需要这种路线，因为未来每个外部能力都必须先经过同样的边界审计。


## MCP 协议的核心思路

MCP（Model Context Protocol）的核心思路是把 AI 工具的连接标准化。在没有 MCP 之前，每个 AI 工具需要自己写适配器去连接飞书、GitHub、Google Drive 等服务。有了 MCP 之后，工具只需要实现 MCP 协议，就可以被任何支持 MCP 的 AI 客户端调用。

这对个人工作流的意义在于：它让工具编排变成可组合的。你可以用一个 AI 客户端同时调用飞书消息、GitHub PR 和本地文件系统，而不需要为每个组合写定制代码。但这也带来了风险：工具越多，权限面越大，审计越难。

## ADK Agent 的边界设计

ADK（Agent Development Kit）Agent 是比 MCP 更高一层的抽象。它不只是连接工具，还定义了 Agent 的行为边界：Agent 可以做什么、不可以做什么、做了之后需要谁确认。在古月浮屿的语境里，ADK Agent 对应的是"灯塔建议 -> 审核队列 -> 人工确认"这条流程。

具体来说，一个 ADK Agent 接入飞书 MCP 后，它可以：
1. 读取公开消息（需要授权 token）
2. 生成回复草案（进入审核队列）
3. 建议创建日历事件（进入审核队列）
4. 不能直接发送消息（动作边界）
5. 不能读取私密文档（数据边界）

这些边界不是在 prompt 里写的软约束，而是在代码层面通过权限模型强制的。ADK Agent 的 `run()` 方法在执行任何动作前都会检查权限策略，不通过就拒绝。

## 从工具实验到世界治理

飞书 MCP 和 ADK Agent 的实验不只是一个技术探索，它还在为世界治理积累经验。当未来 WorldOS 需要接入更多外部能力时——比如 GitHub Actions、Vercel 部署、AI Provider——同样的边界设计会反复出现：身份边界、动作边界、证据边界。

这个节点的价值在于保留了一条从"工具很酷"到"工具可控"的路线。读者看完后应该理解：AI 工具的成熟不是"能做更多事"，而是"知道什么事不能做"。这和世界设计公理中"AI 是增强层，不是生命层"完全一致。

## 实际接入的经验

在实际接入飞书 MCP 时，几个容易踩的坑值得记录。第一，飞书开放平台的权限粒度比较粗，一个 token 往往能访问比预期更多的数据。解决方式是在 ADK Agent 层面做二次裁剪：即使 token 有权限读取所有文档，Agent 也只读取公开标记的文档。

第二，飞书的消息格式比较复杂（富文本、卡片、交互组件），AI 生成回复时容易产生格式错误。解决方式是让 AI 生成纯文本草案，由人工确认后再用飞书 SDK 转换为富文本格式。这保证了 AI 不会直接发送格式错误的消息。

第三，飞书的事件回调需要服务端接收，但当前阶段只有本地 LAN 环境。解决方式是用飞书的 webhook 模式（主动拉取而非被动推送），或者暂时不接入实时事件，只用主动查询模式。这种降级不丢功能，只丢实时性。

## 与世界协议的对齐

飞书 MCP 节点最终要回答的问题是：外部能力如何服从世界协议。世界协议要求所有外部能力经过三层审计：身份审计（谁能代表谁行动）、动作审计（能做什么不能做什么）、证据审计（做了什么留下了什么记录）。

飞书 MCP 只是一个具体案例。未来接入 GitHub、Vercel、AI Provider 时，同样的三层审计会重复出现。这个节点的价值在于保留了一个完整的审计案例，让未来的接入有参照标准。这也是为什么它被纳入"技术与 AI 探索路径"——它不只是工具教程，而是世界治理的一部分。

## 本地阶段的降级策略

当前阶段以本地 LAN 为主，飞书 MCP 的实时事件回调无法在没有公网入口的环境中运行。降级策略是使用主动查询模式代替被动推送：Agent 定时拉取飞书消息，而不是等待 webhook 触发。这种降级保留了核心功能（读取消息、生成草案），只牺牲了实时性。

降级不等于功能缺失。在本地环境中，Agent 仍然可以读取公开消息、生成回复草案、建议日历事件。区别在于这些操作由人工触发而非自动响应。这种"人工触发 + AI 草拟 + 人工确认"的模式，恰好与世界的审核队列设计一致——AI 建议，人决定。

当未来世界上线后，可以切换为 webhook 模式获得实时性。但边界设计不需要改变：身份审计、动作审计、证据审计三层在本地和线上完全一致。这也是"本地成熟后再上线"原则的体现——边界在本地阶段就定义好，上线只是切换运行环境。

## 权限模型与可见性映射

飞书 MCP 的权限粒度需要与 WorldOS 的可见性体系对齐。WorldOS 定义了 8 种可见性：public、semiPublic、private、family、partner、vault、sealed、silent。飞书 MCP 连接器在读取数据时，必须遵守这套可见性体系——只返回 `public` 和 `semiPublic` 的内容给 AI Agent，其余可见性的内容由服务端直接过滤，不经过 MCP 协议传输。

这种映射确保了 AI Agent 即使有飞书 token 的读取权限，也无法获取私密内容。权限不是在 prompt 里写的软约束，而是在数据访问层硬编码的过滤条件。飞书 MCP 只是一个具体案例，但权限映射的模式适用于所有外部连接器：连接器的权限不能大于世界协议的权限。

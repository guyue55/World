# AI 灯塔边界

AI 是灯塔，不是太阳。

它可以：

- 照亮路径
- 整理内容
- 提出草案
- 辅助搜索
- 提醒风险

但它不能：

- 替造物主选择道路
- 自动发布内容
- 读取未授权的私密区域
- 越过 `vault`
- 覆盖原始记录

## 灯塔的职责

灯塔的价值不是替人思考，而是在信息很多、路径很多的时候帮人看见方向。对公开访客来说，它应该回答“我在哪里”“能去哪”“下一步看什么”。对造物主来说，它可以整理草稿、建议关系、提示风险，但这些建议必须停留在审核队列里，不能直接改动世界源文件。

因此，当前灯塔默认运行在低光模式：不接远程模型，不读取私密内容，不写入数据，只基于公开索引、路径和节点摘要提供导览。这个低光模式不是临时降级，而是安全基线。未来即使接入真实 Provider，也必须先证明同样的只读边界仍然成立。

## 为什么不能成为太阳

太阳意味着中心化的照明和生命来源。如果 AI 成为太阳，世界就会逐渐围绕它的输出旋转：内容为了被模型生成而写，关系为了被模型推荐而连，权限为了方便回答而松动。这和古月浮屿的初衷相反。世界应该先能自然生长，AI 只是增强层。

边界必须由后端和数据契约控制，而不是靠前端隐藏按钮。前端可以展示“你看不到这里”，但不能决定“你是否有权看到这里”。AI 也一样：它只能接收后端允许投喂的上下文，不能自己去翻私密层、保险箱或封存记录。

## 未来接入的条件

真实 AI Provider 接入之前，需要满足三个条件。第一，服务端拥有明确的上下文裁剪器，只允许公开或授权内容进入模型请求。第二，所有建议都有 `not-executed`、`requiresReview`、`humanRequired` 等审核状态。第三，检查脚本能证明没有客户端 API Key、没有自动发布、没有删除或改权限入口。

只要这些条件没有完成，灯塔就保持低光只读。它仍然有用：帮助访客进入地图、路径和档案馆；帮助维护者发现可整理的方向；也提醒我们，真正成熟的 AI 能力不是“会说话”，而是知道自己不能做什么。


## 低光模式的技术实现

低光模式不是"AI 功能未完成"的占位符，而是一个独立完整的产品形态。在低光模式下，灯塔页面展示预设的三个导览问题（我在哪、能去哪、下一步看什么）、基于公开数据的静态路径推荐、以及核心节点列表。这些内容全部由 `getLighthouseRecommendedNodes()` 和 `getLighthouseRecommendedPaths()` 从公开索引中生成，不需要任何 AI API 调用。

当真实 AI Provider 接入后，低光模式仍然作为降级方案保留。`isLighthouseAvailable()` 函数检查 API Key 是否存在、AI 端点是否可达。任何一个条件不满足，灯塔自动切换到低光模式。这意味着 AI 涨价、限流、停服或网络故障时，世界仍然可以正常导览。

## 上下文裁剪器

AI 灯塔的核心安全机制是上下文裁剪器。它不是在 prompt 里告诉 AI"请不要读取私密内容"——这种软约束可以被 prompt injection 绕过。真正的裁剪发生在数据层：`getPublicWorldObjectIndex()` 只返回 `visibility === 'public'` 的节点、路径和事件，AI 组件拿到的数据已经被裁剪过。

这种"数据先于 prompt"的边界策略，意味着即使 AI 被注入恶意指令，它也无法访问私密层——因为数据根本不在它的上下文里。这是"后端控制权限，前端控制体现"原则在 AI 边界的具体实现。

## 审核队列设计

AI 生成的所有建议——无论是路径推荐、内容摘要还是关系建议——都进入审核队列，不直接修改世界源文件。审核队列的状态包括 `not-executed`（未执行）、`requiresReview`（待审核）、`humanRequired`（需人工确认）和 `rejected`（已拒绝）。

审核队列目前是本地概念，由 `data/world-kernel/worldos-local-owner-workbench-contract-v1.json` 定义。未来如果接入真实 AI，审核队列会变成 API 端点，但原则不变：AI 建议 -> 审核队列 -> 人工确认 -> 才能写入。AI 永远不直接写入世界源文件。

## 灯塔的三个导览问题

灯塔页面（/ask）以三个核心问题作为导览起点：

1. **我在哪？** —— 回答访客当前在世界中的位置。这不仅是 URL 层面的位置，更是概念层面的位置：你在公开世界层，可以进入地图、时间流或档案馆。
2. **我能去哪？** —— 给出当前位置的出口选项。如果你在首页，建议先走 8 分钟路径；如果你在技术节点，推荐相关技术节点；如果你在路径终点，推荐下一条路径。
3. **下一步看什么？** —— 基于公开数据推荐具体节点。推荐逻辑由 `getLighthouseRecommendedNodes()` 实现，优先推荐有 featured 标记的代表节点和与当前内容相关的节点。

这三个问题不是静态的 FAQ，而是动态的导览逻辑。在低光模式下，它们由预设答案回答。在 AI 模式下，它们由 AI 基于公开上下文生成回答。但无论哪种模式，答案的数据源都是 `getPublicWorldObjectIndex()`——只读公开数据。

## 灯塔与健康度

灯塔页面还承担世界健康度展示的职责。`LighthouseStatus` 组件展示当前世界的公开节点数、路径数、关系数和事件数。这些数字让访客知道世界不是静态页面，而是一个正在生长的系统。

如果某些功能处于低光模式（AI 不可用、某些区域内容较少），灯塔会如实展示状态，而不是假装一切正常。这种诚实是"前台浪漫、后台清醒"原则的延伸：前台可以漂亮，但不能撒谎。

## 灯塔与档案馆的分工

灯塔和档案馆都提供"找到内容"的能力，但方式不同。灯塔是"我不知道我要找什么，帮我看看"——它提供导览问题、路径推荐和核心节点列表。档案馆是"我知道我要找什么，帮我找到"——它提供搜索框、筛选器和结果列表。

这种分工对应了两种访客心态：探索心态和检索心态。探索心态的访客走灯塔 -> 路径 -> 节点的路线；检索心态的访客走档案馆 -> 搜索结果 -> 节点的路线。两条路线最终都到达节点详情页，但入口不同。

在低光模式下，灯塔退回为三个静态导览问题和预设推荐。但档案馆不需要降级——它本来就是静态的搜索功能，不依赖 AI。这保证了即使 AI 完全不可用，访客仍然可以通过档案馆找到任何公开内容。世界不依赖灯塔是否点亮。

# 命令行工具设计原则

好的 CLI 工具像一把称手的刀。它不该需要说明书，拿起来就能用。

## 核心原则

1. **默认行为合理**：不给参数也应该能工作
2. **输出可管道**：stdout 输出数据，stderr 输出日志
3. **退出码有意义**：0 成功，非 0 失败
4. **帮助即时可用**：`-h` 或 `--help` 永远有效

## 命名约定

- 短选项用单字母：`-v`
- 长选项用完整词：`--verbose`
- 子命令用动词：`git commit`、`npm install`

## 配置优先级

命令行参数 > 环境变量 > 配置文件 > 默认值

## 错误信息

好的错误信息告诉用户发生了什么、为什么、怎么办。不要只输出 `Error: failed`。


## 进阶实践

好的 CLI 工具还应该支持 shell 补全。通过 `--completion` 子命令生成 bash/zsh/fish 补全脚本，让用户输入命令时事半功倍。颜色输出也要考虑：终端支持时用彩色高亮关键信息，管道模式下自动关闭颜色。日志级别从静默到调试，让用户自己选择需要多少信息。

# 世界数据守门人

世界可以浪漫，但数据必须清醒。

V1 使用 Zod 和脚本校验：

```bash
npm run validate:world
npm run check:public
```

守门规则包括：

- public 节点必须有 summary
- path 不能引用不存在的节点
- relation 不能引用不存在的节点
- private / vault 不进入公开构建
- AI 未审核内容不可公开

## 守门人守的不是格式

数据校验表面上是在检查字段，实际上是在守住世界的可信度。一个节点缺少 summary，读者就不知道为什么要点进去；路径引用不存在的 slug，旅程就会断；关系没有说明，星线就会变成装饰；私密内容进入公开构建，则直接破坏世界的根基。

所以守门人不是“开发时烦人的脚本”，而是世界的边界感。它让每一次新增内容都必须回答：这个东西能不能被公开？它属于哪个区域？有没有正文？能从哪条路径进入？读完之后能去哪里？这些问题看似琐碎，却决定了本地世界是成熟产品，还是一堆松散文件。

## 本地成熟度的新职责

进入阶段 9 之后，守门人不再只检查能不能构建，还要检查好不好逛。`check:local-product-maturity` 会把真实访客路径、LAN 浏览器截图、动态世界入口、节点返回出口、内容和关系密度放到同一个门槛里。它证明的不是公网发布，而是本地世界已经经得起真实使用。

阶段 10 还需要继续提高正文厚度。过去的节点很多只是种子句，适合搭骨架，但不适合长期阅读。内容增长门禁会先盯住新手路径和代表节点，让第一次进入的人不再只看到一句宣言，而能读到完整解释、边界和下一步。

## 清醒后台，浪漫前台

前台可以像星河，后台必须像账本。每个 JSON 契约、每个检查脚本、每份 RC 报告都在回答同一个问题：这个世界是不是诚实？如果当前只完成本地 LAN，就不要伪装生产上线；如果 AI 只是低光导览，就不要暗示它能读私密；如果内容还薄，就要承认并补厚。

数据守门人的最终目标，是让浪漫不靠运气维持。只要后台足够清醒，前台才能放心地柔软、漂亮、可探索。


## 门禁体系的全貌

古月浮屿的门禁体系不是一组散落的脚本，而是一个分层结构：

- **日常层**：`check:daily`（主线 + lint + typecheck）、`check:boundary`（API + 脚本 + 权限 + 命令脊柱）
- **内容层**：`check:content`（密度 + 内容生命 + 本地增长）、`check:worldos-content-density`（节点/路径/关系/事件数量和正文长度）
- **发布层**：`check:rc:fast`（RC 快检）、`release:local-rc`（完整本地 RC：build + smoke + LAN + audit）
- **治理层**：`check:mainline`（主线治理 + 产品发布 + 公开体验 + 成熟度 + API 边界 + 脚本 + 命令脊柱 + 运行时）

每一层都有对应的契约文件定义阈值。比如内容层的 `worldos-content-life-contract-v1.json` 定义了 `minPublicNodeContentCharacters: 300`，发布层的 `worldos-1-external-evidence-template-v1.json` 定义了 K5-external 的阻断项。门禁不是"跑一下看看通不通过"，而是"契约定义了什么是准备好，门禁验证是否满足契约"。

## 数据校验的具体实现

`validate:world` 脚本用 Zod schema 校验所有 JSON 数据文件的结构。它检查：节点必须有 id/slug/title/summary/areaId/visibility/contentPath；路径的 nodeSlugs 必须引用存在的公开节点；关系的 from/to 必须引用存在的节点；事件的 nodeIds/areaIds 必须引用存在的实体。

`check:public` 脚本验证公开构建产物。它检查：`public/world-index.json` 和 `public/world-manifest.json` 存在且可解析；公开路由的 HTML 产物存在；私密路由不出现在公开产物中。这些校验保证了构建产物与数据源的一致性。

## 守门人的进化方向

当前守门人主要做"有没有"的检查：有没有 summary、有没有 contentPath、有没有关系。未来需要进化到"好不好"的检查：summary 是否足够清晰、正文是否有可读结构、关系是否真的有语义关联。

这种进化需要更智能的校验逻辑——不是简单的字段存在性检查，而是内容质量的量化评估。但在当前阶段，"有没有"的检查已经足够有价值，因为它防止了最常见的问题：空节点、断路径、孤立内容。守门人会随着世界一起成熟。

## 门禁驱动的开发流程

古月浮屿的开发流程是"门禁驱动"的：不是先开发再补门禁，而是先定义门禁再开发。每一轮 RC 都在收紧门禁——RC2 收紧了内容密度，RC3 收紧了主线治理，RC5 收紧了 API 边界，RC6 收紧了命令脊柱，RC8 收紧了 LAN RC 验收。

这种模式的好处是：门禁一旦设定，就自动防止退步。比如内容密度门禁从 30 字符提升到 300 字符后，任何新节点如果正文不足 300 字符，`check:worldos-content-density` 会立即报错。开发者不需要记住"正文要写够"，门禁会替他记住。

门禁驱动的代价是：初期开发速度较慢，因为每个改动都需要通过多层门禁。但长期来看，它节省了大量的调试和回滚时间——因为问题在门禁阶段就被发现，而不是在生产环境被用户发现。

## 守门人与世界诚实

守门人的最终目标是保证世界的诚实。`productionLive: false` 意味着"我们还没有真实上线"，这个状态由 `check:product-release` 验证。如果有人偷偷改成 true 但没有完成 K5-external 验收，门禁会报错。

这种"状态与证据绑定"的设计，让世界不会撒谎。状态布尔值不是可以随意修改的字符串，而是由门禁守护的事实声明。要改变状态，必须先通过对应的门禁。这是"后台清醒"原则的终极体现：不是信任开发者会诚实，而是用门禁确保诚实。

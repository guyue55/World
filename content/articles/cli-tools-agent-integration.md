# CLI 工具接入 Agent

这篇节点把 Codex、Claude Code、Gemini CLI、MCP、Skills 和自定义模型接入经验放在一起看。

核心经验不是“哪个工具最强”，而是要形成一套可复用工作流：

- 统一 API 与模型路由要能解释错误来源。
- CLI 工具要保留可回滚配置。
- Skills / 插件只做增强，不要变成不可审计黑箱。
- 长任务必须有阶段计划、门禁和中间状态提示。
- 失败信息要沉淀为节点，而不是散落在聊天里。

它属于技术星域，也会和 AI 灯塔、创世台、世界维护流程相连。

## 工作流的核心理念

工具会换代，但工作流不会。今天用 Codex，明天可能换别的。重要的是把"统一路由、可回滚配置、可审计插件、阶段计划、失败沉淀"这套方法论固定下来，让它不依赖任何一个具体工具。

## 配置管理示例

```bash
# 备份当前 CLI 配置（可回滚）
cp ~/.codex/config.json ~/.codex/config.json.bak

# 恢复配置
cp ~/.codex/config.json.bak ~/.codex/config.json
```

每次修改 CLI 配置前先备份，出问题可以秒级回滚。这条规则看似简单，但在多工具协作时特别重要——当 Codex 和 Gemini CLI 共享同一个 API Key 但配置格式不同时，改一个可能影响另一个。配置文件版本化管理（用 Git 跟踪 `~/.codex/` 目录）是更彻底的方案，但至少要做到改前备份。

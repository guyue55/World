# 古月浮屿的数字宇宙定义

古月浮屿所说的数字宇宙，不是公共多人 3D 元宇宙，也不是虚拟经济空间。

它是一个个人数字世界：

- 以内容为星体
- 以区域为空间
- 以关系为星线
- 以时间为河流
- 以权限为边界
- 以规则为自然法则
- 以 AI 为灯塔
- 以导出和传承为未来根系

## 更准确的定义

个人数字宇宙是一套把个人经验结构化的系统。它不是为了制造宏大的概念，而是为了让生活、项目、知识和记忆不再散落在不同工具里。一个想法进入这里之后，会被问几个朴素的问题：它属于哪里？和谁有关？现在处于什么生命阶段？适合公开吗？读完之后应该去哪里？

这些问题让内容拥有位置感。位置感比分类更重要，因为分类只是抽屉，位置则能形成路线。技术记录可以连接到项目复盘，项目复盘可以连接到规则，规则又会影响后续工具和 AI 的边界。世界感就来自这种可追踪的互相影响。

## 和知识库的区别

知识库强调存取效率，个人数字宇宙还强调体验和边界。它既要让自己能维护，也要让访客能进入。对外，它必须像一个清澈的网站：入口明确、文案中文优先、路径不迷路。对内，它又必须像一个操作系统：数据有协议，权限有事实源，构建和发布有门禁。

因此，古月浮屿不是把 Notion、博客、相册、项目管理和 AI 聊天简单拼起来。它更像一层世界协议，把不同来源的内容翻译成节点、区域、关系、事件和路径。前台浪漫，后台清醒，这是它能长期运行的基础。

## 为什么先做本地成熟

当前阶段只要求 localhost 和 LAN IP 可访问，是因为世界还在打磨成熟度。外部部署会带来域名、HTTPS、缓存、线上监控等问题，但这些问题不应该遮住更核心的产品问题：第一次来的人能不能逛懂？公开内容是否足够真实？动态效果是否服务理解？权限边界是否可靠？

当本地世界已经能稳定回答这些问题，再谈外部发布才有意义。成熟不是把旗子插到公网，而是在本地就已经经得起真实阅读、真实巡检和重复维护。


## 宇宙的组成元素

个人数字宇宙由七种基本元素构成，每种元素都有明确的职责：

- **Node（节点）**：世界的最小内容单位，可以是文章、项目、记忆、规则或碎片。每个节点有护照（标题、摘要、区域、生命周期、权限）。
- **Area（区域）**：节点的空间归属，相当于星座。八个一级区域覆盖创世原点、技术星域、产品工坊、灵感云层、记忆湖、时间河、档案馆和 AI 灯塔。
- **Relation（关系）**：节点之间的星线，有类型（派生、主题、引用）和强度。强关系必须有人工 review 和 note 说明。
- **Path（路径）**：精选的节点序列，告诉访客"按什么顺序看"。每条路径有目标受众和预估阅读时间。
- **Event（事件）**：世界线上的时间点，记录世界如何生长。事件关联节点和区域，形成可回望的时间线。
- **Permission（权限）**：从 public 到 sealed 的可见性光谱。权限由后端事实源控制，前端只负责展示。
- **WorldState（世界状态）**：全局布尔值和门禁状态，记录世界当前的成熟度。

这些元素不是抽象概念，而是 JSON 文件中的真实字段。`data/domains/experience/nodes.json` 是节点的唯一事实源，`data/core/relations.json` 是关系的唯一事实源。所有页面都从这些文件读取数据，不存在第二份数据源。

## 从"发布"到"安放"

传统内容平台的核心动作是"发布"：写完一篇文章，点发布，它出现在时间线上，然后被新文章覆盖。个人数字宇宙的核心动作是"安放"：写完一个节点，给它分配区域、补充护照、连接关系、放入路径，然后它在世界的拓扑结构中获得一个位置。

安放比发布慢，但安放让内容有了位置感。一个被安放的节点不会因为时间流逝而消失在时间线底部，它始终在地图上的某个区域里，始终可以通过关系星线被发现，始终在路径中被推荐给合适的访客。这就是为什么古月浮屿要求每个公开节点必须有 summary、worldTitle、contentPath 和至少一条关系——这些不是格式要求，而是安放要求。

## 世界的可迁移性

个人数字宇宙不依赖任何特定平台。当前用 Next.js 实现，但所有内容都是 Markdown 和 JSON 文件，数据结构是自描述的。如果有一天 Next.js 不再合适，内容文件可以原封不动地迁移到 Astro、Quartz 或任何其他框架。节点结构、区域定义、关系数据和路径配置都是纯 JSON，不包含任何框架特定代码。

这种可迁移性是"生命档案"属性的基石。一个真正属于个人的数字世界，不应该被绑死在某个云服务或某个框架上。内容是世界的灵魂，框架只是世界当前的身体。身体可以换，灵魂不能丢。

## 从概念到产品

个人数字宇宙的概念很容易理解，但把它做成产品需要回答很多具体问题：节点正文用什么格式（Markdown）；数据用什么结构（JSON）；页面怎么渲染（Next.js SSG）；权限怎么控制（middleware 服务端拦截）；AI 怎么接入（只读公开索引 + 低光降级）；内容怎么维护（低心力四档 + 修复日）；世界怎么导出（一键 zip）；未来怎么传承（传承协议）。

每一个选择都不是随意的。用 Markdown 而不是富文本编辑器，是因为 Markdown 可迁移、可 diff、不依赖平台。用 JSON 而不是数据库，是因为 JSON 是文件、可 git 版本控制、可静态构建。用 Next.js SSG 而不是 SPA，是因为 SSG 生成的 HTML 对 SEO 和首屏性能友好。用 middleware 而不是前端判断权限，是因为前端隐藏可以被绕过。

这些选择共同构成了世界的工程基座。基座稳定了，内容才能安心生长。如果基座频繁变动——换框架、换数据格式、换权限模型——内容维护就会变成跟随技术债务的追赶，而不是安放自己的生活。

## 世界的健康度指标

一个健康的个人数字宇宙可以用几个指标衡量：公开节点是否有足够正文（>= 400 字符）；每个节点是否至少有 2 条关系星线；每个一级区域是否有代表节点；公开路径是否覆盖所有区域；核心节点是否有足够深度（>= 2000 字符）；内容密度门禁是否通过。

这些指标不是 KPI，而是世界的体检报告。`check:content` 命令一次性输出所有指标的状态。当某个指标下降时——比如新增节点忘记加关系——门禁会立即报错。这种自动化体检让世界不需要人工巡查就能保持健康。

## 世界的节奏与仪式

个人数字宇宙不是全天候运行的系统，它有自己的节奏和仪式。每周的修复日是世界的"清尘时间"：不新增概念，只整理已有内容。每月的月报是世界的"回望时间"：看这个月长了什么、什么在沉睡。每季的季报是世界的"校准时间"：检查方向是否偏离初衷。每年的年度回望是世界的"年轮"：记录一整年的生长轨迹。

这些仪式不是形式主义。它们的存在是因为个人世界最大的敌人不是技术难题，而是维护疲劳。如果没有节奏，世界要么被过度维护（变成全职工作），要么被忽视（变成废弃仓库）。节奏让世界以可持续的速度生长——不需要每天投入，但也不会沉默太久。

低心力维护流程是节奏的执行层。1 分钟记一粒光，5 分钟整理一颗种子，15 分钟连几条星线，30 分钟修一个小区域。这四档覆盖了从"今天只有碎片时间"到"周末有一小时"的所有场景。世界不需要大块时间才能维护，它需要的是持续的小动作。
